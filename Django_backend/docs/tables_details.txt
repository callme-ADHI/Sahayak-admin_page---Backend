================================================================================
JOB KARO – DJANGO BACKEND DATABASE DOCUMENTATION
================================================================================
Version        : 2.0.0 (Django Edition)
Date           : 2026-03-04
Framework      : Django 6.0 + Django REST Framework
Database       : PostgreSQL (production) / SQLite (development)
ORM            : Django ORM (replaces Supabase PostgREST)
Auth           : JWT via djangorestframework-simplejwt
Migration Tool : Django Migrations (replaces raw SQL scripts)

This document is the complete, authoritative reference for the Job Karo
Django backend database schema. It covers all tables, columns, types,
constraints, indexes, and architectural decisions.

────────────────────────────────────────────────
TABLE: users
Django Model: accounts.User
Migration: accounts/migrations/0001_initial.py
────────────────────────────────────────────────
Purpose:
  Central identity entity for ALL platform participants (customers, workers,
  admins). Extends Django's AbstractBaseUser + PermissionsMixin.
  Replaces the manual `users` table from the Supabase schema.

  KEY CHANGE vs Supabase: password_hash column removed.
  Django handles all password hashing internally via PBKDF2.
  ADMINS table removed — replaced by is_staff / is_superuser flags + Groups.

Columns:
  id                  UUID         PK, default=uuid4(), editable=False
                                   Matches original schema column name.

  phone               VARCHAR(15)  UNIQUE, NOT NULL, db_index=True
                                   PRIMARY LOGIN CREDENTIAL.
                                   Django's USERNAME_FIELD is set to 'phone'.

  email               VARCHAR(255) UNIQUE, NULL, db_index=True
                                   Optional. Used as fallback login.

  password            TEXT         NOT NULL (managed by Django AbstractBaseUser)
                                   Stored as PBKDF2 hash — never plaintext.
                                   Column name in DB: 'password'.

  account_status      VARCHAR(20)  NOT NULL, DEFAULT='active', db_index=True
                                   Choices: active | suspended | banned

  is_verified         BOOLEAN      DEFAULT=False
                                   KYC verification flag.

  is_active           BOOLEAN      DEFAULT=True   (Django auth: login allowed)
  is_staff            BOOLEAN      DEFAULT=False  (Django admin access)
  is_superuser        BOOLEAN      DEFAULT=False  (all permissions bypass)

  is_deleted          BOOLEAN      DEFAULT=False, db_index=True
                                   Soft delete flag. Never hard-delete users.
  deleted_at          TIMESTAMP    NULL — populated when soft-deleted.

  created_at          TIMESTAMP    DEFAULT=NOW()  (db_column = 'created_at')
                                   Django field: date_joined
  updated_at          TIMESTAMP    AUTO — updated on every save.
  last_login          TIMESTAMP    NULL — managed by Django auth on login.

Indexes:
  idx_users_phone     → phone
  idx_users_email     → email
  idx_users_status    → account_status
  idx_users_deleted   → is_deleted

Constraints:
  phone UNIQUE
  email UNIQUE (when not null)

Auth Backend: accounts.backends.PhoneOrEmailBackend
  → Supports login via phone OR email + password.
  → Soft-deleted/banned users are rejected at auth layer.

SECTION 4: FINANCE  (App: finance)
================================================================================

IMPORTANT: ALL monetary fields use DecimalField(max_digits=12, decimal_places=2).
Float is NEVER used for money. Binary float rounding (e.g. 99.9999999) is
unacceptable in any financial system.

────────────────────────────────────────────────
TABLE: wallets
Django Model: finance.Wallet
────────────────────────────────────────────────
Purpose:
  Per-user platform credit/escrow wallet.
  CHANGE: wallet_balance removed from users table.
  This is the single source of truth for user balances.

  CONCURRENCY RULE: ALL balance updates MUST use:
    Wallet.objects.select_for_update().get(user=user)
  inside an atomic transaction block to prevent race conditions.

Columns:
  user_id         UUID         PK, FK → users.id (CASCADE)
  current_balance DECIMAL(12,2) DEFAULT=0.00, >= 0
  currency        VARCHAR(5)   DEFAULT='INR'
  is_frozen       BOOLEAN      DEFAULT=False
  last_updated_at TIMESTAMP    AUTO
  created_at      TIMESTAMP    AUTO

────────────────────────────────────────────────
TABLE: transactions
Django Model: finance.Transaction
────────────────────────────────────────────────
Purpose:
  Full double-entry ledger of all platform financial movements.
  Every money movement (payment, refund, payout, top-up) creates one record.
  db_column='transaction_id' matches original schema.

Columns:
  transaction_id     UUID         PK (db_column='transaction_id')
  job_id             UUID         FK → jobs.job_id (PROTECT), NULL
                                  on_delete=PROTECT — financial records
                                  must survive job deletion.
  from_user_id       UUID         FK → users.id (PROTECT), NULL
  to_user_id         UUID         FK → users.id (PROTECT), NULL
  transaction_type   VARCHAR(20)  DEFAULT='payment', db_index=True
                                  Choices: payment | refund | payout |
                                           top_up | reversal
  amount             DECIMAL(12,2) NOT NULL, >= 0
  currency           VARCHAR(5)   DEFAULT='INR'
  transaction_status VARCHAR(20)  DEFAULT='pending', db_index=True
                                  Choices: pending | success | failed | reversed
  payment_method     VARCHAR(30)  blank=True — upi | card | wallet | netbanking
  gateway_reference  VARCHAR(255) blank=True
  failure_reason     TEXT         blank=True
  metadata           JSONB        DEFAULT={}
  created_at         TIMESTAMP    AUTO
  updated_at         TIMESTAMP    AUTO

Indexes:
  idx_txn_status → transaction_status
  idx_txn_job    → job_id
  idx_txn_from   → from_user_id
  idx_txn_to     → to_user_id

────────────────────────────────────────────────
TABLE: audit_logs
Django Model: moderation.AuditLog
────────────────────────────────────────────────
Purpose:
  Immutable, append-only system-wide event log for security and compliance.

  CRITICAL DESIGN RULES:
  1. Records CANNOT be modified — model.save() raises PermissionError if pk exists.
  2. Records CANNOT be deleted — model.delete() raises PermissionError always.
  3. Django admin has_change_permission=False and has_delete_permission=False.
  4. No updated_at column — records are immutable.

  This replaces Supabase Row-Level Security audit trail with
  application-level enforcement.

Columns:
  id          UUID     PK
  admin_id    UUID     FK → users.id (SET_NULL), NULL, db_column='admin_id'
                       Must be is_staff=True user.
  action_type VARCHAR(50) NOT NULL, db_index=True
                          e.g. create | update | delete | login | logout | ban
  target_type VARCHAR(50) blank=True, db_index=True
                          e.g. user | job | worker | system
  target_id   UUID     NULL
  description TEXT     blank=True
  ip_address  INET     NULL
  user_agent  TEXT     blank=True
  before_data JSONB    NULL — snapshot before change
  after_data  JSONB    NULL — snapshot after change
  created_at  TIMESTAMP AUTO (immutable)

Indexes:
  idx_al_action → action_type
  idx_al_target → target_type
  idx_al_date   → created_at

SECTION 7: KEY ARCHITECTURAL DIFFERENCES FROM SUPABASE SCHEMA
================================================================================

  ┌────────────────────────────────────────────────────────────────────────────┐
  │  Concern          │  Supabase Schema          │  Django Schema             │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Auth             │  Manual password_hash      │  AbstractBaseUser +        │
  │                   │  column on users table     │  Django PBKDF2 hashing     │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Admin Access     │  Separate ADMINS table     │  is_staff / is_superuser   │
  │                   │  with email/role columns   │  + Django Groups           │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Money Fields     │  FLOAT everywhere          │  DecimalField(12,2) always │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Wallet           │  wallet_balance on users   │  Separate wallets table    │
  │                   │  table                     │  OneToOne(User)            │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Permissions      │  Row-Level Security (RLS)  │  DRF Permission Classes    │
  │                   │  in PostgreSQL             │  + Queryset filtering      │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Timestamps       │  Inconsistent / missing    │  created_at + updated_at   │
  │                   │  on many tables            │  on ALL transactional tbls │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Soft Delete      │  No soft delete            │  is_deleted + deleted_at   │
  │                   │                            │  on users, jobs, workers   │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Audit Trail      │  Supabase audit extension  │  Immutable AuditLog model  │
  │                   │                            │  (save/delete blocked)     │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Coordinates      │  FLOAT lat/lng             │  Decimal(9,6) lat/lng      │
  ├────────────────────────────────────────────────────────────────────────────┤
  │  Job FK naming    │  job_id (PK), skill_id     │  db_column preserves all   │
  │                   │  (FK to skills)            │  original column names     │
  └────────────────────────────────────────────────────────────────────────────┘

SECTION 9: INDEX STRATEGY
================================================================================

All indexes follow the naming pattern: idx_{table_abbreviation}_{field}

Performance indexes:
  idx_users_phone           → phone (primary login field)
  idx_users_email           → email (login fallback)
  idx_users_status          → account_status (filtering)
  idx_users_deleted         → is_deleted (soft delete filter)
  idx_user_roles_active     → (user, is_active)
  idx_vr_status             → verification status
  idx_vr_worker             → worker lookups
  idx_wp_status             → worker_status
  idx_wp_available          → is_available (marketplace matching)
  idx_cat_active            → (is_active, is_deleted)
  idx_cat_name              → name (search)
  idx_addr_user_default     → (user, is_default)
  idx_addr_city             → city (geographic filter)
  idx_jobs_status_date      → (job_status, scheduled_at) — COMPOSITE (hot path)
  idx_jobs_user_status      → (user, job_status) — COMPOSITE (hot path)
  idx_jobs_deleted          → is_deleted
  idx_ja_job_status         → (job, assignment_status) — COMPOSITE
  idx_ja_worker_status      → (worker, assignment_status) — COMPOSITE
  idx_jsh_job               → job (status history lookups)
  idx_wc_composite          → (worker, category) — COMPOSITE
  idx_txn_status            → transaction_status
  idx_txn_job               → job FK
  idx_txn_from / idx_txn_to → sender/receiver lookups
  idx_disputes_status       → dispute status
  idx_disputes_job          → job FK
  idx_reports_status        → report status
  idx_reports_type          → report type
  idx_reports_reporter      → reporter FK
  idx_ban_entity            → (entity_id, entity_type) — COMPOSITE
  idx_ban_active            → is_active
  idx_al_action             → action_type
  idx_al_target             → target_type
  idx_al_date               → created_at (chronological log queries)
  idx_notif_unread          → (is_read, recipient)
  idx_notif_priority        → priority

SECTION 10: MIGRATION COMMANDS
================================================================================

Initial Setup:
  # 1. Set up virtual environment:
  python3 -m venv venv && source venv/bin/activate

  # 2. Install dependencies:
  pip install -r requirements.txt

  # 3. Configure .env (copy .env.example and fill values):
  cp .env.example .env

  # 4. Apply migrations:
  python manage.py migrate

  # 5. Create a superuser for Django admin:
  python manage.py createsuperuser

  # 6. Run dev server:
  python manage.py runserver

Connecting to PostgreSQL (Production):
  Update .env:
    DB_ENGINE=django.db.backends.postgresql
    DB_NAME=your_db_name
    DB_USER=your_db_user
    DB_PASSWORD=your_db_password
    DB_HOST=your_db_host
    DB_PORT=5432

  Then run: python manage.py migrate
